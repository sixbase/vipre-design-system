Quickstart · MSP Menu (SideNav)

Add the menu to your app in ~5 minutes

Works in any framework. The menu is three things — one stylesheet, a block of markup, and one tiny behavior file. Add those three, wire one event, done.

1

Add the stylesheet (once)

This carries the whole look — colors, spacing, the navy rail. Import it one time for your whole app.

/* CSS, or your app's global styles */
@import 'vipre.css';
2

Drop in the markup

Paste the menu's HTML where you want the rail. The classes (vds-sidenav…) are the same in every framework. Grab the full block from the demo files (linked below) — here's the shape:

<nav class="vds-sidenav" aria-label="Product">
  <!-- account header, product cards, footer rows -->
  <!-- copy the block from index.html (link below) -->
</nav>
3

Add the behavior + listen for clicks

One dependency-free file wires collapse, the product accordions, the active row, and the collapsed-rail tooltips. Then you listen for one event — sidenav:select — and send it to your router. Pick your framework:

import { useEffect, useRef } from 'react'
import { initSideNav } from './msp-menu.js'
import './vipre.css'

function AppNav() {
  const ref = useRef(null)
  useEffect(() => {
    const menu = initSideNav(ref.current)                 // wires behavior
    const onSelect = e => router.push(e.detail.id)        // your router
    ref.current.addEventListener('sidenav:select', onSelect)
    return () => { menu.destroy() }                       // cleanup
  }, [])
  // paste the <nav class="vds-sidenav">…</nav> markup here
  return <nav ref={ref} className="vds-sidenav" dangerouslySetInnerHTML={{ __html: MENU }} />
}

Once we publish the React package, this becomes one line — <SideNav sections={data} onSelect={…} /> — no markup or effect needed.

// 1. add vipre.css to "styles" in angular.json (or @import it)
// 2. paste the vds-sidenav markup into the component template, with #nav
import { AfterViewInit, ElementRef, ViewChild } from '@angular/core'
import { initSideNav } from './msp-menu.js'

export class AppNav implements AfterViewInit {
  @ViewChild('nav') nav!: ElementRef<HTMLElement>
  constructor(private router: Router) {}
  ngAfterViewInit() {
    initSideNav(this.nav.nativeElement)
    this.nav.nativeElement.addEventListener('sidenav:select',
      (e: any) => this.router.navigate([e.detail.id]))
  }
}

Same three files, wired in ngAfterViewInit. When we ship Web Components, this becomes <vds-side-nav> with no wiring.

<link rel="stylesheet" href="vipre.css">

<nav class="vds-sidenav" aria-label="Product"> … </nav>

<script type="module" src="msp-menu.js"></script>  <!-- auto-wires -->
<script type="module">
  document.querySelector('.vds-sidenav')
    .addEventListener('sidenav:select', e => location.hash = e.detail.id)
</script>

The script auto-wires every .vds-sidenav on the page. Works for Vue, Svelte, or no framework at all.

✓

What you get vs. what you wire

The menu handles
  • The whole look (light + dark)
  • Collapse / expand + animation
  • Product accordions open/close
  • Active-row highlight
  • Collapsed tooltips + keyboard + a11y
You wire
  • The data (what rows/products show)
  • Where clicks go (sidenav:select → your router)
  • Which products are locked (your plan/entitlements)
  • The account name up top (your logged-in user)

Where does the data come from? Today the rows live in the markup you paste. When the React package ships, you'll pass a single data object instead — sections, account, activeId — and the menu builds itself.
