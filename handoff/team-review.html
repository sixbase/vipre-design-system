One source of truth, two ways to consume it

How the design team maintains the system, and how the React and Angular teams pull it into real apps. The menu (SideNav) is our pilot component — the model below applies to every component after it.

💬 In plain words

Designers make the parts. A machine packs them in a box. Builders take parts from the box — nobody rebuilds a part that already exists.

1

How the design team maintains it

The repo is the single source of truth. Nothing is styled by hand in an app.

  • Tokens are the foundation. Every value lives in _tokens.scss and ships as a --vds-* CSS variable, in three tiers — primitive ramps → semantic intent (--vds-primary) → scale (spacing, radius, motion). Light/dark is a token swap under .dark.
  • Components own their styles. Each is a folder — SideNav.jsx + co-located SideNav.scss (BEM, vds- prefixed) + index.js. Rule: no raw hex/px in a component — if a value is missing, add a token.
  • The docs site is the showroom + spec. Every component has a page: live example, props table, the raw vds- markup, and a11y notes. Fix a component once → every page and every consumer inherits it.
  • Two build outputs are what ship (kept stable):
# the two consumable stylesheets
npm run build:tokens   → dist/vipre-tokens.css  (just the --vds-* variables)
npm run build:styles   → dist/vipre.css         (tokens + every component's CSS)

# package.json exports stay fixed:
"."            → React components (source JSX)
"./tokens.css" "./styles.css"   → the CSS bundles above

Maintenance loop: edit a token or a component's SCSS/JSX → verify on the docs site (light + dark, mobile → desktop) → cut a versioned release. Consumers pick it up on their next update. Decisions + gotchas are logged in the playbook.

💬 In plain words

A “token” is a saved setting with a name — like a paint color called “brand blue.” The team changes it in one place, and everything using it changes too. No copy-paste, no chasing.

2

How the engineering teams consume it

Two tracks, because we run two frameworks. Both pull from the same build outputs above.

React projects

Install & import

Add the package once, import the CSS once, then use components by name. Fixes arrive on version bump.

// once, at the app root
import '@vipre/design-system/styles.css'

// anywhere you need the menu
import { SideNav } from '@vipre/design-system'

<SideNav
  account={acct}
  sections={data}       // your entitlements
  activeId={page}
  onSelect={setPage} />

The app owns the data (routing, entitlements). The look + behavior come from the package.

Angular & anything else

CSS + markup + behavior

Angular can't import a React component. It uses the shared CSS, renders the vds- markup, and wires the tiny behavior file.

/* once, in global styles */
@import '@vipre/design-system/styles.css';

// render the vds-sidenav markup in the template,
// then attach behavior + listen for clicks
import { initSideNav } from '@vipre/.../vanilla'
initSideNav(navEl)
navEl.addEventListener('sidenav:select',
  e => router.navigate(e.detail.id))

This is exactly the handoff/msp-menu folder we already built.

💬 In plain words

React builders just ask for the menu and it appears. Angular builders can’t do that yet, so they use a copy of the menu plus a tiny helper that makes it click.

⚠️ Heads up

Names like @vipre/design-system are pretend for now — the real package doesn’t exist yet. Making it real is decision #3 below.

Recommended end-state Build each component once as a Web Component (<vds-side-nav>). It's browser-native, so React, Angular, and plain HTML all drop in the same element — data in via properties, events out via CustomEvent. We're already halfway: the CSS is shared and the behavior is extracted.

💬 In plain words

A Web Component is a menu the browser itself understands. Build it once, and every kind of builder — React, Angular, plain HTML — uses the very same one.

3

The contract between the two teams

This is the seam. Anything on the left flows to consumers automatically; anything on the right needs a version bump + a heads-up.

ChangeFlows automaticallyBreaking (version + comms)
Token values — a color, spacing, radiusYes—
Internal component stylingYes—
New optional prop / attributeYes—
Renaming a token or vds- class—Breaking
Changing the behavior spec / event names—Breaking
Removing or renaming a prop—Breaking

The stable contract is small: the --vds-* token names, the vds-sidenav__* class names, and four behaviors — toggle --collapsed, toggle the product --reveal--open, set aria-current on the active row, float the collapsed tooltip — plus the sidenav:select event.

💬 In plain words

Left side = tiny changes nobody notices, safe to do anytime. Right side = changes that could break someone’s app, so we tell everyone first.

What we need to lock in

Three decisions and the pipeline is set:

Confirm the frameworks each team ships in — React, Angular, anything else.

Agree to build each component once (Web Components) rather than twice.

Stand up the publish pipeline — a versioned package + the CSS bundles — so consumers update in one line.

The menu is live and clickable, and the handoff/msp-menu folder already shows the Angular-style path (CSS + markup + behavior). Ask Alvin for the demo. This is a direction to align on, not a final spec.
