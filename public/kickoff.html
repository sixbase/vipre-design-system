1

The idea (30 seconds)

Think of a shared toy box. We build each toy once — a button, a menu — and put it in the box. Everyone builds screens from the same box. So everything matches, and when we fix a toy, everyone’s toy gets fixed.

2

Two ways to use a toy

🧰
Better

Take it from the box

Add the box once. Then just ask for a toy. We make it better → yours gets better on its own.

✏️
Okay

Trace it by hand

Copy the toy into your project. Works — but your copy stays old when we fix the real one.

3

Our twist: two kinds of blocks

Some of us build with React blocks, some with Angular blocks. We don’t want to build every toy twice. Lucky us — a toy has two parts:

  • ✓The paint (how it looks) already works on both. Done.
  • ✓The moving part (how it clicks) we build once, so both blocks share it.

Come to the website to look and learn. Take the real toy from the box to use.

4

What to decide — jot notes as we go

A. Where does the “real” version live?

We want the toy box (design system) to be the source of truth — not Figma. Figma becomes for sketching, not the final word.

B. Which blocks does each team use?

React, Angular, or both — per project. This decides how each team takes toys from the box.

C. Do we build each toy once and share it?

Build once so React and Angular use the same one — instead of building everything twice.

D. Who sets up the box, and by when?

Someone needs to package it so teams install it in one line and get updates.
