Meeting guide · Design + Engineering

Handoff & source-of-truth alignment

How we deliver from design → engineering using the design system and the prototypes we build with Claude — and how we move off Figma as the source of truth. Use this to run the room and leave with real decisions.

The one decision everything hangs on: what is the source of truth, and what job does each tool do?

Today Figma is the source of truth and eng pulls specs from it. We want the design system (code + docs) to be the source of truth instead.

If the room agrees to this, the meeting succeeded:

• Design system (code) = source of truth for anything real — tokens, components, states, behavior.
• Figma = ideation, flows, stakeholder review — reflects the system, doesn’t define it.
• Claude prototypes = exploration that gets promoted into the system once validated.

2

Decisions to make in the room

Tagged by who owns the call: Design Eng Both

  • Design  Where do designers work, and what’s the spec of record?

    Figma for exploration — but the coded component becomes the truth, not the Figma frame. Agree on that.

  • Design  How does a design change get into the system?

    A designer can’t merge SCSS. Who translates a proposal into tokens/components? Who reviews and approves?

  • Design  What happens to Figma?

    Retire it, keep it for ideation only, or keep it in sync with tokens? If synced, which direction — ideally code → Figma.

  • Eng  Where do specs come from without Figma?

    Is the docs site (props, tokens, markup, a11y) enough to build from? What’s missing vs. Figma redlines? This is make-or-break.

  • Eng  What does “production-ready” actually mean?

    Pin the bar: versioned package, semver, changelog, tests, TypeScript types, accessibility, visual regression.

  • Eng  Consumption model + frameworks?

    Confirm React vs Angular per team. Build once as Web Components, or ship a React package + CSS/behavior contract for Angular?

  • Eng  The publishing pipeline — the current blocker.

    Where does the package live (private npm / GitHub Packages)? Who stands it up? Version + release cadence? No pipeline = no clean handoff.

  • Both  How does a Claude prototype become production?

    A prototype isn’t production code. Who hardens it — componentizes, tokenizes, tests? What’s the promotion checklist?

  • Both  The contract + governance.

    Token names, class names, component API, events — who owns it, how are breaking changes versioned + announced? Who owns the repo and release cadence?

  • Both  Definition of “handoff done.”

    One shared checklist both sides sign, so “done” isn’t subjective.

3

Don’t leave the room without…

  1. Agreement that the design system is the source of truth (Figma demoted to ideation/flows).
  2. Which frameworks eng ships in — React / Angular / both.
  3. An owner for the design system and an owner + date for the package pipeline.
  4. The Figma-replacement checklist (the table above) with an owner per row.
  5. A rough prototype → production promotion process — who does what at each step.
!

Name these risks out loud

No published package yet — “production-ready consumption” is blocked until it exists. Highest-priority action item.
React + Angular means “build once” needs a real answer (Web Components) or you maintain everything twice.
Don’t yank Figma too fast — designers lose their working surface. Phase it: tokens + components move to code first; flows stay in Figma for now.
Stakeholders who only speak Figma — the docs site + prototype links become the new review surface. Make sure they can use it.

Worth saying in the room: the hard parts are already started — a live docs site as the spec surface, tokens as the shared language, and working prototypes. What’s missing is mostly process + a publish pipeline, not more design.
